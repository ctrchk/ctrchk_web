# 香港城市運輸單車 (CTRC HK) 產品 Backlog

本文件由 **CTRC HK (香港城市運輸單車)** 產品管理團隊編寫。本 Backlog 的核心目標是基於 v2.1.1 Beta 的現有代碼實現，梳理系統當前的功能板塊、性能邊界以及防刷作弊防禦機制。透過本文件，我們將明確未來數月內的開發優先順序，確保平台在進行大規模全港推廣前，具備工業級的穩定性、一致性、安全性與綠色低碳出行的核心吸引力。

---

## 一、 產品現狀概述 (Product Overview)

CTRC HK 目前定位於一個融合「智慧綠色通勤導航」與「遊戲化三軌經濟」的專用單車生態系統。平台在代碼層面實現了高度的雙平台切換分工機制（網頁端與 PWA 獨立應用端），針對高頻單車騎行與智慧出行提供了硬件級別的深度集成。

### 1. CTRC HK 目前擁有的功能與系統架構
1. **雙平台分工運作機制 (Web & PWA Dual Mode)**：
   - **網頁端 (Web Mode)**：面向公共搜尋引擎與一般瀏覽者。側重品牌理念宣導、單車生活網誌閱讀與投稿、開放式單車徑路線列表預覽（無置灰或等級鎖定限制），以及高級會員的 GPX 軌跡文件下載。
   - **PWA 應用端 (PWA Mode)**：當用戶將網站「加至主畫面」並以 standalone 獨立應用模式啟動時，系統會自動解鎖底部浮動玻璃導航欄（Liquid Glass Navigation）、進入「沉浸式工業風儀表板 (Industrial HUD)」、啟用屏幕 Wake Lock 鎖定，解鎖多人房間實時騎行工作台、動態導航以及每日簽到和提醒功能。
2. **三軌制車手經濟體系 (The Three-Track Economy)**：
   - **成長軌 (Level/EXP)**：支持 50+ 等級稱號晉升。設有不同的騎行模式倍率（旅遊模式 1.5 倍、通勤/DRT 模式 1.0 倍、自由模式 0.8 倍）。每次完成騎行結算會觸發「多巴胺隨機盲盒」，概率獲得額外的 $+1\sim10\text{ XP}$ 獎勵。
   - **實力軌 (Rolling Mileage)**：基於近 365 天滾動總里程進行每日卡級更新。分為銅卡（0km+ 基礎特權）、銀卡（150km+ 解鎖單車泊位/自訂 5 個途經點/+5% 里程幣）、金卡（500km+ 解鎖 3D 車道巡航/極簡黑金主題/+15% 里程幣）。一年前的舊里程每日按天扣除。
   - **貨幣軌 (Coins)**：騎行結算、簽到和升級禮包可產生里程幣。里程幣可用於在商城解鎖特定的「夜間挑戰」或「特別版」路線，或花費 `100 幣` 在個人中心修復意外中斷的通勤連勝紀錄。
3. **安全反作弊檢測系統 (Anti-Cheat Engine)**：
   - 後端對上傳的 GPS 軌跡進行雙重過濾：限制瞬時/平均時速不得高於 $45\text{ km/h}$（防止開車/地鐵刷里程）；過濾低於 $0.2\text{ km}$ 的極短行程，防止靜止掛機。

### 2. 已完成模組 (Completed Modules)
* **電子錢包集成 (Wallet Pass Integration)**：完美對接了 WalletWallet API 永久免費通道，避開 Apple 開發者帳號年費。支持根據用戶的滾動里程級別，動態生成自適應配色（金黃色 `#F0D372`、科技銀 `#D1D9DF`、古典銅 `#D8A56B`）的 Wallet 卡包。
* **高對比度黑金主題 (Black Gold Theme)**：針對金卡用戶，支持烈日下高可視度的 `--rank-gold` 主題。通過強硬的 `!important` 樣式覆蓋，確保在烈日曝曬下車手依然能清晰辨識導航與 HUD 面板中的關鍵數據。
* **多人房間系統 (Multiplayer Room System)**：取代了舊版複選框，支持用戶開設「公開」或帶「密碼」的私密房間。房主可選擇路線、鎖定進度，並在 Leaflet/MapLibre 地圖上實時繪製和同步隊友的頭像與坐標。
* **斷點續傳機制 (Session Resumption)**：如果多人或單人騎行中途意外斷電、瀏覽器崩潰或網絡異常，本機 `localStorage` 與數據庫 `active_rides` 會保存 `roomCode`。用戶重新啟動 App 後，主頁會彈出「未完成行程卡片」，點擊一鍵即可恢復原有進度與位置同步。

### 3. 可用模組 (Usable Modules)
* **智慧單車導航與報站 (Smart Navigation & TTS)**：基於 Brouter 專用算法提供「避開馬路、專用單車徑優先」的路由。靠近站點 50 米內觸發本地 Web Push 推送或語音報站，終點 8 米內自動結束並觸發數據結算。
* **單車路線圖籍 (Route Catalog)**：全港路線以「未征服（置灰）」和「已通關（彩色）」進行地圖式收集。集齊所有路線可一鍵領取 **10,000 XP** 與 **1,000 里程幣** 超大獎勵包。
* **3D Achievement Badges (3D 徽章展廳)**：在任務中心內，用戶解鎖的各項成就和徽章可通過 `<model-viewer>` 技術，直接加載服務器 `/model/glb` 和 `/model/usdz` 的 3D 物理質感模型，支持在手機屏幕上進行縮放、旋轉。

### 4. 實驗性模組 (Experimental Modules)
* **討論區與社群論壇 (Forum & Topic Replies)**：現有的 `forum.html` 實現了基礎的帖子發佈、評論回覆以及 Like 按鈕，但在實時推播、富文本、舉報過濾與性能方面仍處於初步階段，缺乏精細的主題板塊分類與車隊徽章聯動。
* **單車網誌投稿審核 (Blog Submissions)**：允許一般用戶投稿，管理員在後台進行一鍵審核發佈，但審核流程比較單一，且缺乏投稿被拒絕後的反饋機制與通知。

---

### 5. 絕不可修改的模組 (Never-to-be-changed Modules)
有些核心架構、算法規則與限制是 CTRC HK 賴以維持公平性與穩定性的基石，在未來的迭代中**絕不允許進行結構性修改**：
1. **反作弊的核心速度閥值 ($45\text{ km/h}$ 最大限速)**：
   - 任何人以大於 $45\text{ km/h}$ 的均速或瞬時速度騎行，該次行程必須作廢且不予寫入數據庫，並觸發幽默的警告文案，維護全港排行榜誠信。
2. **0.2 公里最低記錄門檻 (Minimum Distance Filter)**：
   - 拒絕記錄任何小於 $0.2\text{ km}$ 的騎行，防止用戶在家掛機或在原地輕微搖晃設備刷積分。
3. **365 天滾動里程計劃算法 (Rolling Mileage Formula)**：
   - 等級必須每日根據近 365 天（一年前的今天起）的數據滾動計算，不允許改為累積終身里程，以確保車手保持長期持續運動的習慣。
4. **960 路線排除規則 (Route 960 Exclusion)**：
   - 960 號路線（及 `tko-960`）作為最長的終極觀光挑戰路線，明文禁止計入全港挑戰賽 (HK Challenge) 合格資格，且自由騎行模式無法觸發挑戰賽。此安全保護代碼不可更改。

## 二、 優先級劃分 (Backlog Prioritization)

為保證 CTRC HK 的平穩上線，我們將 Backlog 按照 P0（致命/關鍵）、P1（非常重要）、P2（未來改進）和 P3（長期技術戰略）四個維度進行細分。

---

### P0: 關鍵與致命性問題 (Critical / Blockers)
*必須在公開推廣、商用發佈和線下活動前徹底解決。核心關注點：性能、內存洩漏、GPS 精度、視覺一致性、防刷防作弊。*

#### 任務 P0-1: 導航頁 (`nav.html`) 地圖跟隨與 GPS 羅盤快速抖動優化
* **當前狀態**：實時導航使用 `navigator.geolocation.watchPosition` 監聽位置。在城市高密度建築群（如將軍澳市中心、康城高樓群）中，GPS 經緯度及設備指南針 (Heading) 容易出現高頻微幅抖動。
* **面臨問題**：在低速行駛（例如等紅綠燈或低於 3km/h 慢行）時，地圖隨指南針瘋狂旋轉，導致用戶眩暈、無法辨識前進方向。另外，地圖自動鎖定跟隨用戶時，手動平移操作會頻繁被強制拉回。
* **為何重要**：這直接關係到騎行時的物理安全。不穩定的地圖朝向會干擾判斷，導致分心看屏幕而發生交通碰撞。
* **建議實現方案**：
  - 引入**卡爾曼濾波算法 (Kalman Filter)** 或指數滑動平均濾波對 GPS 坐標進行平滑。
  - **低速鎖定補丁**：當速度低於 3km/h 時，地圖軸承 (Bearing) 自動鎖定在當前路網的幾何矢量切線方向上，禁止讀取羅盤 Heading 數據。
  - **手動暫停跟隨**：用戶平移/縮放地圖時，設置 10 秒計時器，暫停強制回中，10 秒無操作或點擊「跟隨我」指南針按鈕後再行恢復。
* **依賴關係**：`nav.html`, `js/main.js`
* **開發難度**：中等 (Medium)
* **估計工作量**：3 人工天
* **優先級**：P0
* **預期影響**：大幅提高極端樓宇環境下的地圖穩定性，杜絕靜止或低速時地圖的劇烈抖動。

#### 任務 P0-2: 騎行工作台 (`ride.html`) 內存洩漏與 Leaflet 瓦片重載管理
* **當前狀態**：PWA Standalone 模式下的騎行頁面在長時間行進時，需要實時在 Leaflet 地圖上繪製 dashed breadcrumb（虛線麵包屑軌跡）。
* **面臨問題**：在長途騎行（如 30km 以上，持續 2 小時以上）過程中，由於 DOM 節點堆積、路徑坐標點緩存未釋放，以及 Leaflet 瓦片地圖在弱網環境下反覆失敗重試，導致嚴重的內存洩漏 (Memory Leak)，這會觸發 iOS standalone 模式自動強制關閉 (Crash)。
* **為何重要**：用戶辛辛苦苦騎行了 30 公里，在臨近終點時 PWA 應用突然閃退，導致數據丟失，會造成極大的用戶負評。
* **建議實現方案**：
  - 對追蹤軌跡坐標進行**道格拉斯-普克算法 (Douglas-Peucker)** 抽稀，每 10 秒只保留關鍵控制點，防止坐標點數無上限膨脹。
  - 定期調用 Leaflet 的 `map.removeLayer()` 清除視野外的歷史瓦片。
  - 將未提交的軌跡即時以 Append 形式增量寫入 `localStorage` 的 `unsyncedTempCoordinates`，確保即使意外閃退也能完全恢復。
* **依賴關係**：`ride.html`, `js/pwa.js`
* **開發難度**：高等 (High)
* **估計工作量**：5 人工天
* **優先級**：P0
* **預期影響**：消除 App 連續運行時的崩潰隱患，內存佔用穩定控制在 120MB 以下。

#### 任務 P0-3: 全局 CSS 變量一致性與黑金主題樣式覆蓋加固
* **當前狀態**：CTRC 使用基於 CSS 變量的 Light / Dark / Gold 三套主題。在 Gold 黑金主題下，系統應對關鍵 UI 進行強高對比度覆蓋。
* **面臨問題**：在實際真機（特別是 iOS Standalone 或烈日室外）測試中，部分新合併的按鈕（如多人房創建、已省車資卡片、自訂途經點）未嚴格遵循全局變量（如 `--hud-accent`, `--hud-bg`），導致出現白底白字、黑底黑字的不可讀色差情況。
* **為何重要**：UI 的對比度缺失會使車手在戶外騎行時需要花費更多時間和精力去辨認屏幕，嚴重威脅騎行安全。
* **建議實現方案**：
  - 對 `css/main.css` 進行系統性審查，確保所有頁面交互組件（包括聊天、論壇、自訂途經點面板）均使用 CSS 變量定義背景與文字。
  - 對 `body.rank-gold` 進行嚴格的黑金高對比度規則覆蓋，強制落實黑底金字（金黃色 `#F0D372` 主體，純黑 `#000000` 文字/邊框）。
* **依賴關係**：`css/main.css`, 所有 HTML 文件
* **開發難度**：低等 (Low)
* **估計工作量**：2 人工天
* **優先級**：P0
* **預期影響**：實現 100% 的 UI 視覺一致性，在 100,000 勒克斯的強烈陽光直射下依然保持極佳的辨識度。

#### 任務 P0-4: PWA 離線緩存同步容錯與 Service Worker 重載機制
* **當前狀態**：PWA 的 `sw.js` 提供了靜態和 API 代理緩存，支持離線記錄騎行並在聯網後自動同步上傳。
* **面臨問題**：在地下隧道、單車徑盲區等頻繁切換網絡的環境下，Service Worker 容易出現緩存卡死。離線騎行歷史在網絡恢復時觸發並行上傳，由於缺乏互斥鎖鎖定，會發生 API 重複請求，導致同一條騎行歷史在 Neon 數據庫中被重複插入兩次以上，從而嚴重干擾排行榜。
* **為何重要**：數據庫數據重複會直接摧毀排行榜和三軌經濟體系的公平性。
* **建議實現方案**：
  - 在前端離線同步代碼中加入 **遞歸式隊列同步鎖 (Sync Mutex Lock)**，同一時間只允許上傳一條未同步記錄。
  - 為每條本地騎行記錄生成一個隨機且唯一的 UUID 寫入數據庫（`client_uuid`），後端 `api/getHistory.js` 對此欄位做唯一性限制 (Unique Constraint)，若重複直接拒絕寫入。
* **依賴關係**：`sw.js`, `api/getHistory.js`, `js/pwa.js`
* **開發難度**：高等 (High)
* **估計工作量**：4 人工天
* **優先級**：P0
* **預期影響**：保證離線數據 100% 不丟失、不重複，弱網/無網環境切換時系統無感過渡。

#### 任務 P0-5: 反作弊系統瞬時速度與軌跡重疊算法加固
* **當前狀態**：後端對騎行歷史記錄限制最大均速為 $45\text{ km/h}$，並有基礎的 anti-bus 防護。
* **面臨問題**：高級作弊用戶可通過第三方工具偽造 GPX 文件並上載，只要均速控制在 40km/h 以下即可輕鬆繞過限速。此外，GPS 抖動也可能導致瞬時速度突變超過 45km/h，將誠實的車手誤判為作弊。
* **為何重要**：若排行榜被作弊刷分佔領，將徹底摧毀平台的聲譽、打擊真實用戶的騎行積極性。
* **建議實現方案**：
  - **瞬時速度平滑處理**：後端在解析 GeoJSON 或坐標數組時，不只看單點速度，改為採用滑動窗口（例如每 5 個點）計算移動均速，過濾因 GPS 漂移造成的瞬時超速誤判。
  - **軌跡擬合檢測**：引入簡化的**單車徑路網覆蓋率算法 (Map-matching Coverage)**。計算上傳的軌跡點中，距離全港單車徑（`CYCTRACK` 圖層）50米之內的比例。若比例低於 60%（且非自由模式），判定為公路開車或公共交通作弊。
* **依賴關係**：`api/getHistory.js`, `api/user.js`
* **開發難度**：高等 (High)
* **估計工作量**：4 人工天
* **優先級**：P0
* **預期影響**：作弊攔截率提升至 95% 以上，同時將誠實用戶的誤判率降至 0.5% 以下。

### P1: 非常重要問題 (Very Important)
*未來數月內必須交付的模組，核心關注點：里程卡價值、電子錢包卡面實時刷新、任務系統完善、個人中心體驗與雙平台無縫引導。*

#### 任務 P1-1: 里程卡未來 30 天過期里程可視化與保級壓力預警功能
* **當前狀態**：里程卡（銅/銀/金）基於 365 天滾動里程。`mileage.html` 展示了卡級，但用戶難以預知自己的里程何時會過期。
* **面臨問題**：用戶在沒有任何警示的情況下，可能在某天起床時發現里程卡突然從金卡降為銀卡，只因一年前的今天他騎行了 100km，這會對用戶造成負面的情緒衝擊（失落與不滿）。
* **為何重要**：可視化的過期預警能轉化為強大的上癮激勵。告知用戶「您在 7 天內有 25km 里程即將過期，再騎行 10km 即可穩固金卡身份」，能極大刺激用戶再次出發。
* **建議實現方案**：
  - 在 `mileage.html` 新增「里程過期預警圖表（線形圖或條形圖）」，展示未來 30 天內每日即將失效的里程數額。
  - 後端 API 在返回用戶里程數據時，計算 `approaching_expiry_km` (未來 30 天即將失效的里程累計) 並返回給前端。
* **依賴關係**：`mileage.html`, `api/user.js`
* **開發難度**：中等 (Medium)
* **估計工作量**：3 人工天
* **優先級**：P1
* **預期影響**：顯著提升用戶活躍度與保級留存率，將「被動扣里程」變為主動的「騎行拉動機制」。

#### 任務 P1-2: Apple & Google Wallet 卡包數據實時刷新與動態 Push 機制
* **當前狀態**：用戶可在 `mileage.html` 通過 WalletWallet 接口生成 Apple/Google Wallet 卡包。但卡包在手機內是不會自動刷新的靜態卡，用戶每次升級或騎行後，卡面上的里程依然是舊數據。
* **面臨問題**：如果需要手動刪卡重加，電子錢包卡包的體驗就會大打折扣，淪為一次性玩具。
* **為何重要**：實時更新的 Wallet 憑證可作為車手隨身的「榮譽勳章」，在線下車友聚會、商戶兌換或日常通勤中出示時，提供極高的尊榮感。
* **建議實現方案**：
  - 啟用 WalletWallet 的 **WebService 刷新協議**。
  - 當用戶成功在後端 `api/getHistory.js` 提交新騎行記錄並導致等級/總里程/卡級發生變更時，後端異步向 WalletWallet API 發送更新推送（Push Update）請求，由 WalletWallet 服務端向 iOS/Android 用戶的手機卡包發送實時刷新信號。
* **依賴關係**：`api/user.js`, `api/getHistory.js`
* **開發難度**：高等 (High)
* **估計工作量**：4 人工天
* **優先級**：P1
* **預期影響**：實現手機電子錢包卡包數據的秒級異步自動同步刷新，卡包實用性提高 200%。

#### 任務 P1-3: 任務系統每週任務正式落地與動態配置數據庫化
* **當前狀態**：`tasks.html` 中的「每週任務」目前處於置灰置盲的置留狀態（顯示「🚧 即將推出」），每日任務則為純前端硬編碼（`DAILY_TASKS` 數組）。
* **面臨問題**：運營人員無法根據節假日或特別活動靈活配置或發佈新的任務（例如「端午節騎行海濱線送雙倍里程幣」），必須手動修改前端代碼重新部署。
* **為何重要**：豐富、變化的任務是保持中長期用戶粘性（D30+ 留存）的核心動力，每週任務能激勵用戶進行中長途（如 20km 以上）騎行。
* **建議實現方案**：
  - 激活數據庫 `task_definitions` 表，完全廢除前端硬編碼任務。
  - 後端 `api/user.js` 新增 `action=get-tasks` 接口，從數據庫動態讀取當前處於 `is_active = true` 狀態的每日與每週任務。
  - 前端 `tasks.html` 調用此接口渲染。在每週一凌晨自動結算每週任務（本週騎行 20km、骑行3條不同路線等），完成後支持一鍵領取里程幣。
* **依賴關係**：`tasks.html`, `database-schema.sql`, `api/user.js`
* **開發難度**：中等 (Medium)
* **估計工作量**：4 人工天
* **優先級**：P1
* **預期影響**：任務系統完全實現動態後台化運營，用戶每週騎行頻次預計可提高 25%。

#### 任務 P1-4: 用戶註冊與個人資料修補流程（Onboarding）閉環優化
* **當前狀態**：新用戶通過註冊頁註冊後，需要填寫經驗、單車類型和首選地區等以升級至「高級會員（Senior）」。
* **面臨問題**：目前從註冊完成、電郵驗證到 profile-setup 的跳轉缺乏流暢的導引，導致新用戶卡在半路，未填寫完檔案，無法享受 GPX 下載或銀卡多站自訂導航等核心高級權益。
* **為何重要**：新用戶的 Onboarding 轉化率直接決定了推廣時的資金和時間回報率。
* **建議實現方案**：
  - **引導式向導 (Interactive Wizard)**：在註冊完畢並在 `verify-email.html` 驗證成功後，頁面不直接回首頁，而是強制展示精美的 3 步個人資料設定向導。
  - 設置首頁「高级特權未鎖定」的頂部常駐卡片，點擊引導去 `profile-setup.html`。
  - 上傳頭像組件加入前端圖片裁剪（Crop）壓縮處理，防止用戶上傳原始大圖造成 Vercel 帶寬過載。
* **依賴關係**：`register.html`, `verify-email.html`, `profile-setup.html`, `js/main.js`
* **開發難度**：中等 (Medium)
* **估計工作量**：3 人工天
* **優先級**：P1
* **預期影響**：註冊到完整個人資料（高級會員）的轉化率提升至 80% 以上。

### P2: 未來功能 (Future Features)
*不屬於核心導航和結算的社交與輔助工具，需在核心功能穩定後（預計未來數月）逐步在代碼中進行細化。*

#### 任務 P2-1: 論壇系統 (`forum.html`) 板塊細分與熱門話題動態聚合

* **面臨問題**：隨著用戶增長，所有地區（將軍澳、沙田、港島）和類別（二手交易、單車維修、車隊召集）的話題堆疊在一起，查找困難，且缺乏板塊管理員（Moderators）防灌水審核。
* **為何重要**：健康的社群論壇能沉澱高質量的用戶 UGC 內容，為平台帶來穩定的自然搜索流量 (SEO)。
* **建議實現方案**：
  - 根據數據庫中的 `tag` 字段，在前端 `forum.html` 實現分類選項卡：【全部】、【路線討論】、【車商驛站】、【裝備交易】、【車友召集】。
  - 實現「精華帖 (Featured Topic)」置頂功能，管理員可在後台一鍵將高質量路線攻略置頂。
* **依賴關係**：`forum.html`, `forum_topic.html`, `api/forum.js`
* **開發難度**：中等 (Medium)
* **估計工作量**：3 人工天
* **優先級**：P2
* **預期影響**：論壇可讀性大幅增加，促進用戶騎行之餘的文字交流與互動。

#### 任務 P2-2: 單車網誌 (`blog.html`) 投稿與管理後台雙向審核機制優化
* **當前狀態**：系統允許一般用戶投稿，保存在數據庫中，`published` 默認為 false，等管理員手動修改。
* **面臨問題**：用戶投稿後不知道進度，管理員在 `admin.html` 裡也缺乏便捷的可視化審核、修改和回覆界面，導致投稿功能被閒置。
* **為何重要**：利用全港車手的投稿，能低成本擴充高質量的騎行遊記和路況反饋網誌。
* **建議實現方案**：
  - 在 `admin_badges.html` 或獨立管理頁面中新增「網誌審核模塊」，支持在線預覽投稿 HTML 內容，一鍵點擊【批准發佈】、【拒絕並通知退回】、【編輯內容】。
  - 用戶可在個人中心 `profile.html` 的「我的投稿」板塊看到「審核中」、「已發佈」、「退回重修」等狀態以及管理員的退回意見。
* **依賴關係**：`blog.html`, `admin.html`, `api/blog.js`
* **開發難度**：中等 (Medium)
* **估計工作量**：3 人工天
* **優先級**：P2
* **預期影響**：打通 UCG 網誌內容的審核閉環，提高社群創作者投稿的熱情。

#### 任務 P2-3: 房間制多人聯動系統 (`ride_rooms`) 即時心跳同步與網絡抖動容錯
* **當前狀態**：`ride_rooms` 提供實時多人騎行，各成員的位置通過定時 `heartbeat` API 上傳和下載。
* **面臨問題**：在單車徑（尤其是郊外和沿海信號不穩定處），API 經常偶發性超時。若某個隊友因網絡信號暫時中斷 30 秒，他的地圖頭像會立即從地圖上消失，或者出現瞬移、漂移，用戶體驗不夠平滑。
* **為何重要**：多人騎行的核心價值在於「同屏破風」的陪伴感與競技樂趣，地圖上隊友的頻繁消失和卡頓會極大影響社交體驗。
* **建議實現方案**：
  - **位置插值算法 (Dead Reckoning)**：前端在繪製隊友頭像時，若 heartbeat 未能如期抵達（如超時），不立即刪除頭像，而是根據該隊友最後一次上報的速度和方向，自動在地圖上平滑預測前進，直至 2 分鐘完全失聯後再將其置灰或隱藏。
  - 弱網絡包重試機制：heartbeat 失敗時進行指數退避重試，防止網絡雪崩。
* **依賴關係**：`ride.html`, `api/user.js`
* **開發難度**：高等 (High)
* **估計工作量**：4 人工天
* **優先級**：P2
* **預期影響**：多人騎行位置同屏渲染平滑度大幅提高，完全屏蔽弱網環境造成的頭像卡頓與閃爍。

### P3: 長期願景 (Long-term Vision)
*涉及重大技術重構、重資本運營或跨界商戶聯動。核心關注點：離線計算、全單頁應用 SPA 改造、實體跨界合作。*

#### 任務 P3-1: 離線導航與 WASM 本地路網拓撲路徑規劃
* **當前狀態**：目前的導航 `nav.html` 依賴在線 Brouter 或備用 OSRM API 服務。
* **面臨問題**：在香港部分郊野山路、海濱死角或漫長隧道中，移動數據網絡信號可能完全丟失。此時如果用戶偏離了路線，導航將無法進行偏航重算，甚至地圖一片空白，導致用戶在野外迷失方向。
* **為何重要**：極限環境下的安全保障是 CTRC 作為專業級導航的核心賣點，離線功能是其實現質變的一步。
* **建議實現方案**：
  - 通過 WebAssembly (WASM) 技術，將精簡版的 Brouter C++/Java 核心引擎編譯並嵌入到瀏覽器 PWA 內核中。
  - 將全港單車徑路網（`tko_tracks.json` 及全港拓撲數據，壓縮至 5MB 以內）寫入 PWA 的 IndexedDB 數據庫中。
  - 當檢測到斷網時，自動切換至本機 WASM 規劃引擎，實現 100% 離線路由重算。
* **依賴關係**：`nav.html`, `sw.js`
* **開發難度**：超高 (Very High)
* **估計工作量**：15 人工天
* **優先級**：P3
* **預期影響**：平台具備全天候、無盲區的完全離線單車道導航能力，確立行業技術領先地位。

#### 任務 P3-2: 全網頁單頁應用 (SPA - Single Page Application) 深度遷移
* **當前狀態**：CTRC 目前通過 `js/main.js` 來實現部分的頁面動態加載、初始化以及 `pwa-page-show` 事件監聽。但本質上依然是基於傳統多 HTML 文件跳轉的「偽 SPA」架構。
* **面臨問題**：每次跳轉 HTML（如從 `index` 到 `ride`），瀏覽器都會重新初始化部分靜態資源、CSS 樣式以及 JS 狀態，造成短暫的黑屏、白屏過渡，不符合現代原生應用的極致流暢體感。
* **為何重要**：為了在未來將 CTRC HK 封裝成 native 應用（iOS App Store / Google Play），必須擁有真正無感刷新、狀態常駐的全單頁 SPA 架構。
* **建議實現方案**：
  - 基於原生 JS 或者是輕量路由庫（如 Navigo 或 Page.js），將所有 HTML 頁面徹底重構成統一在 `index.html` 的一個 `<div id="app-viewport">` 內進行組件化切換。
  - 全局常駐地圖實例 (Singleton Map Instance)，頁面切換時不銷毀 Leaflet 地圖，僅在 DOM 中移動地圖容器並調用 `map.invalidateSize()`，實現地圖跨頁面秒開。
* **依賴關係**：全部 HTML 與 JS 文件的重構
* **開發難度**：高等 (High)
* **估計工作量**：12 人工天
* **優先級**：P3
* **預期影響**：頁面切換實現 0 延遲與電影級轉場動畫，完美符合 Native-like PWA 技術指標。

#### 任務 P3-3: 實體商戶與低碳零售里程幣兌換 API 接口設計
* **當前狀態**：用戶騎行產生的里程幣（Coins）目前只能在線上購買解鎖特別路線，用途相對單一。
* **面臨問題**：虛擬幣如果沒有實體權益支撐，中後期用戶的收集意願會大幅降低，導致低碳減排的激勵模型失效。
* **為何重要**：通過與香港本地實體商戶（如單車租賃店、海濱咖啡館、便利店）合作，實現「騎行省錢、綠色消費」的閉環，將能極大引爆大眾群體。
* **建議實現方案**：
  - 設計開放的 **CTRC Merchant API** 機制。
  - 允許合作商戶通過安全 Token 接入 CTRC。用戶在商戶出示 Wallet 卡包二維碼（內含車手安全 ID），商戶使用掃描槍或專用網頁輸入，即可實時調用接口扣除用戶指定的里程幣數額，並在商戶處兌換實體商品（如免費美式咖啡、租車 9 折優惠、單車配件）。
* **依賴關係**：`api/user.js`, 新數據庫表 `merchant_vouchers`
* **開發難度**：中等 (Medium)
* **估計工作量**：6 人工天
* **優先級**：P3
* **預期影響**：打通線上與線下（O2O）低碳綠色經濟閉環，商業變現與合作想象空間無限。

### 1. 七月：核心穩定與安全保固 (P0 衝刺月)
* **開發目標**：解決所有阻礙平台推廣的安全和崩潰隱患，提供極致平穩的實時騎行體驗。
* **詳細里程碑**：
  - **第 1-2 週**：完成 **任務 P0-1 (地圖跟隨與低速鎖定)** 與 **任務 P0-3 (黑金主題樣式補丁)**。在將軍澳天晉、康城高樓區進行 20 次真機實測，保證低速行駛時地圖無旋轉抖動。
  - **第 3 週**：攻克 **任務 P0-2 (Leaflet 內存洩漏)**。引入道格拉斯-普克抽稀算法，並確保在 iPhone 12/13 等舊設備上連續導航 3 小時不閃退。
  - **第 4 週**：完成 **任務 P0-4 (離線同步隊列鎖)** 與 **任務 P0-5 (反作弊擬合檢測算法)**。保障數據一致性，杜絕後台刷單。

### 2. 八月：任務落地與用戶留存 (P1 攻堅月)
* **開發目標**：盤活平台的三軌制遊戲化激勵機制，引導新用戶順暢轉化，提升 D15 與 D30 留存指標。
* **詳細里程碑**：
  - **第 1-2 週**：完成 **任務 P1-1 (30天過期里程壓力預警)** 與 **任務 P1-4 (引導式註冊向導)**。在 mileage 頁面完成預估圖表的動態渲染。
  - **第 3-4 週**：完成 **任務 P1-2 (電子錢包卡面實時秒級自動更新刷新)** 與 **任務 P1-3 (數據庫每週任務上線)**。運營人員可在後台動態創建和更替任務，車手的手機卡包在騎行結束後異步完成秒級刷新。

### 3. 九月：社交社群與網誌內容 (P2 建設月)
* **開發目標**：豐富平台的社群交互屬性，讓用戶在不騎行時也能在平台上進行高質量的交流與創作分享。
* **詳細里程碑**：
  - **第 1-2 週**：落實 **任務 P2-1 (論壇板塊分類與置頂精華)**，論壇實現有序、分板塊的健康管理。
  - **第 3 週**：落實 **任務 P2-2 (網誌投稿雙向審核流)**，打通車友創作遊記、管理員編輯發佈的 O2O UCG 機制。
  - **第 4 週**：升級 **任務 P2-3 (多人房插值與網絡容錯算法)**，即使在信號惡劣的將軍澳跨灣大橋下或隧道口，隊友的實時位置頭像也絕不卡頓和瞬移。

### 4. 十月：技術儲備 — 全網頁 SPA 單頁應用遷移 (P3 技術戰略)
* **開發目標**：重構前端架構，為未來的原生 Android / iOS 殼封裝鋪平道路。
* **詳細里程碑**：
  - **第 1-2 週**：搭建 Page.js 單頁路由體系，設計統一的 App Shell 和動態視口容器。
  - **第 3-4 週**：重構地圖模塊，實現全局單例地圖實例 (Singleton Map Instance) 常駐內存。將 `index`, `dashboard`, `ride`, `nav` 等多個 HTML 合併為輕量組件。頁面切換實現無縫電影級轉場動畫，徹底消除白屏。

### 5. 十一月：技術儲備 — 完全離線導航路網 (P3 技術戰略)
* **開發目標**：在無網絡信號下，依然提供 100% 可信賴的單車道級導航與偏航重規劃服務。
* **詳細里程碑**：
  - **第 1-2 週**：使用 Emscripten 將 Brouter C++ 核心引擎編譯為 WebAssembly。
  - **第 3-4 週**：將全港單車徑（`CYCTRACK`）及拓撲數據進行深度 Gzip 壓縮（控制在 4MB 內），並開發離線偏離自動路由重算算法，網絡斷開時無縫切換至本地 WASM 規劃。

### 6. 十二月：O2O 綠色消費生態商戶對接 (P3 商業變現)
* **開發目標**：打通里程幣的實體消費閉環，實現「騎行減碳、消費省錢、商戶引流」的多元共贏綠色單車生態。
* **詳細里程碑**：
  - **第 1-2 週**：完成 **任務 P3-3 (Merchant API)** 及其憑證表的設計與安全加固。
  - **第 3-4 週**：發佈商戶自主入駐與核銷控制台（支持手機相機直接掃描用戶手機 Wallet 卡包中的動態二維條碼）。在將軍澳海濱和北角碼頭挑選 5 家合作商戶開展線下兌換 Beta 測試，正式宣佈 CTRC HK 進入商業化閉環。

## 四、 「絕不應開發的功能」 (Things that should NOT be developed)

為避免軟體工程中常見的 **「功能蔓延 (Feature Creep)」** 和 **「不必要複雜性 (Over-engineering)」**，我們必須在產品藍圖中設置清晰的防火牆，以下功能板塊在 CTRC HK 的中長期規劃中**明確禁止開發**：

### 1. 禁止引入任何「非單車專屬」的導航模式
* **原因**：市場上已有極度成熟的 Google 地圖、高德地圖和 Apple 地圖，它們在機動車導航、公共交通換乘領域擁有絕對壁壘。
* **邊界**：CTRC 必須且只能專注於**香港單車徑優先**的導航。任何關於開車路線規劃、地鐵換乘時刻表、巴士站點實時到站預報等功能，一律禁止開發，保持導航引擎的純粹與輕量。

### 2. 禁止開發複雜的實時音視頻聊天系統
* **原因**：實時音視頻（RTC）技術（如 WebRTC 語音房、實時車隊語音）需要消耗極高的帶寬與服務器成本，且在單車行進中會引入嚴重的風噪干擾，且會極大消耗車手的手機電量，得不償失。
* **邊界**：多人房僅維持實時位置同屏共享、以及 `chat.html` 內非實時的文本聊天。禁止在應用內開發實時語音、實時視頻或車隊語音對講。車友可自行使用已非常普及的 Discord、WhatsApp 或電話對講功能。

### 3. 禁止引入高耗能的 3D 實時全景 AR 導航
* **原因**：雖然 AR 實景導航或 3D 全景地圖（如實時渲染 3D 樹木、車輛與動態天氣）非常酷，但它會導致手機芯片瘋狂過熱、電量崩潰，甚至在夏日烈陽曝曬下導致手機高溫自動關機，威脅行車安全。
* **邊界**：地圖展示維持現有的 Leaflet 2D 主體以及金卡尊享的 Mapbox 輕量 3D 靜態建築物巡航（基於矢量高度）。禁止進行任何實時物理模擬、動態粒子特效和高頻 AR 相機渲染。

### 4. 禁止開發底層電子貨幣或去中心化區塊鏈 (Web3) 錢包
* **原因**：引入去中心化代幣（如 Token 鑄造、鏈上交易、合約質押）會引入極大的金融法律合規風險（如證監會監管、反洗錢審查、防範黑客攻擊），且極易讓綠色通勤變味為金融投機。
* **邊界**：里程幣 (Coins) 必須且僅維持作為**平台內部發行的中心化虛擬點數**。所有的消費和扣減均通過 CTRC 中心化數據庫 API 進行，不允許與任何區塊鏈、虛擬貨幣或鏈上錢包（如 MetaMask）做技術集成。

### 5. 禁止開發通用電商或自營實體物流平台
* **原因**：自建電商平台需要龐大的倉儲、供應鏈、售後客服與物流配送體系，這會使產品管理團隊精力嚴重分散，偏離核心技術。
* **邊界**：里程幣實體兌換一律採用 **「O2O 券碼核銷模式」**，即 CTRC 僅負責發放核銷券碼，由用戶在線下第三方商戶（如單車店、咖啡館）出示卡包進行現場兌換，物資與售後由合作商戶自行承擔。平台絕不經手實體物流。
